## Original request

Let's rearrage the project.

I want to install all the software and adjust all the settings using a separate script, not via docker machinery, so it will be possible to launch the script on macos/ubuntu via bash, and to achieve everything that we have here.

So the script launches bash to install brew, brew installs `nu`. `nu` launches install script to install all the else and to adjust settings.

In future: I want to be able to install all the settings into a separate `xdg_config`directory - to enable this path on some existing working machine.

## Rephrased

Rearrange the project so that all software install and settings configuration happens in a standalone script (not in Docker layers). The script must run on macOS or Ubuntu via plain `bash` and reproduce everything the current Dockerfile produces. Flow: bash bootstrap installs Homebrew → `brew install nushell` → Nushell takes over and runs the rest of the installer and applies settings. The Dockerfile becomes a thin wrapper that calls the same script.

Future goal: support installing the configs into an alternative `XDG_CONFIG_HOME` (or a more general prefix) so the script can be tried on a host that already has its own working configs, without overwriting them.

## Current state (what the Dockerfile does today)

Roughly six concerns are mixed together in `Dockerfile`:

1. **Docker-only glue** — base image, apt proxy, `/etc/gitconfig` fallback, `pbcopy` OSC52 shim, `/etc/sandbox-persistent.sh` env exports, `COPY` layers, `USER` switching.
2. **System deps** — apt: `procps file gcc libc6-dev`.
3. **Brew + brew tools** — Homebrew, then `nushell fzf lazygit helix zellij broot git-delta visidata bat topiary fd jj git-lfs`.
4. **Modules** — vendor copy from `vendor/` (default) or `git clone` (with `--build-arg MODULES_SOURCE=clone`) into `~/repos/`; copy `sandbox-toolkit/` into `~/repos/cozy/sandbox-toolkit/`; copy autoload `.nu` into `~/.config/nushell/autoload/`.
5. **Settings deployment** — `nu -c 'use toolkit.nu; toolkit push-to-machine --force --create-dirs --docker --commit-changes'` from vendored dotfiles; append `docker-files/global-claude.md` to `~/.claude/CLAUDE.md`; copy skills into `~/.claude/skills/`; XDG `~/.config/git/{config,ignore}`; broot defaults.
6. **Late layer** — `topiary install` (clone grammar, build `.so`), `claude` install via `claude.ai/install.sh`, register `nushell` MCP server.

Concerns 2–5 are what the user wants in the portable script. Concerns 1 and 6 (well, parts of 6) need a decision about boundary.

The infra is already partly there: `sandbox-toolkit/install/` has modules for `claude`, `topiary`, `nushell` (build-from-source), `polars`, `rust`, `zellij`, `nu-plugin-image` — but the brew-install line and module-deployment line are inline in the Dockerfile, not in any of those modules.

## Proposed plan

### Layout

```
bootstrap.sh                       # bash entry — installs brew, brew installs nu, hands off to nu
sandbox-toolkit/
  install/
    bootstrap.nu                   # NEW: 'install all' orchestrator (brew tools, modules, dotfiles, skills, ...)
    brew-tools.nu                  # NEW: brew install <list>; idempotent
    modules.nu                     # NEW: clone-from-github OR copy-from-local for ~/repos/*
    dotfiles.nu                    # NEW: invoke vendored dotfiles toolkit push-to-machine
    skills.nu                      # NEW: deploy ~/.claude/skills/
    git-xdg.nu                     # NEW: write ~/.config/git/{config,ignore}
    claude.nu                      # exists — also registers nushell MCP
    topiary.nu                     # exists
    nushell.nu, polars.nu, ...     # exist; not part of default 'all'
```

`bootstrap.sh` is small (~30 lines). All real work lives in nu modules so logic stays structured-data-friendly and re-usable.

### Step ordering inside `bootstrap.nu`

1. `brew-tools install` (uses Linuxbrew/Homebrew already on PATH)
2. `git-xdg apply`
3. `modules fetch` — clone (or rsync from `--local`) into `~/repos/`
4. `dotfiles deploy` — runs `toolkit push-to-machine --force --create-dirs --commit-changes` (drops `--docker` if not in Docker)
5. `skills deploy`
6. Append `global-claude.md` to `~/.claude/CLAUDE.md` (Docker-only? or always? — open question)
7. `topiary install`
8. `claude install` + MCP register (gated by `--with-claude`)

Each sub-step idempotent and prints a one-line summary.

### Dockerfile after the move

The Dockerfile shrinks to: base image → root: apt deps + pbcopy + apt proxy + `/etc/gitconfig` fallback → user agent: `COPY` repo, `RUN bash bootstrap.sh --in-docker --modules=vendor --with-claude` → late env vars and `/etc/sandbox-persistent.sh`. Build args become `bootstrap.sh` flags.

### Future: alternative XDG prefix (`xdg_config` dir)

Introduce a single env var, e.g. `COZY_TARGET_HOME` (defaults to `$HOME`). All install steps that write to `~/.config/...` or `~/.claude/...` resolve under `$COZY_TARGET_HOME` instead. Concretely:

- `git-xdg.nu`, `skills.nu`, `dotfiles.nu`, `topiary.nu`, `claude.nu` accept the prefix.
- Vendored `dotfiles/toolkit.nu` already iterates over `paths-default.csv` paths starting with `~/.config/...`, `~/.claude/...`, `~/.X` — adding a `--target-home` flag there is the main lift.
- `~/repos/` does *not* move (modules are code, not config) unless we also want a `COZY_REPOS_DIR`.
- The user activates the alt prefix on their existing machine by setting `XDG_CONFIG_HOME` (and friends) before launching nushell, OR by sourcing a generated env file from the alt prefix.

This is a follow-up phase, not part of the first refactor.

### Suggested PR sequence (small, reviewable)

1. Extract `brew-tools.nu`, `modules.nu`, `dotfiles.nu`, `skills.nu`, `git-xdg.nu` from inline Dockerfile RUN lines. Wire into `bootstrap.nu`. Dockerfile still works (calls `bootstrap.nu`).
2. Add `bootstrap.sh` — bash entry that brew-installs nu and hands off to `bootstrap.nu`. Verify on Ubuntu (in a clean container) and macOS.
3. Slim the Dockerfile to call `bootstrap.sh`.
4. (Future) `--target-home` / `COZY_TARGET_HOME` plumbing.

## Open questions

Please answer inline (or anywhere — I'll pick them up).

1. **Entry script location & name** — `cozy/bootstrap.sh` at repo root, or `cozy/sandbox-toolkit/install/bootstrap.sh`, or different?

2. **Module fetching on a host** — when running on a user's macOS/Ubuntu (not Docker), should `modules.nu` always `git clone` from GitHub, or also support a local-sibling mode like `toolkit/vendor.nu --local` (rsync from sibling repos)?

3. **Default scope of `bootstrap.nu` "all"** — should the host installer include the heavy build-from-source items (`install/nushell.nu`, `install/zellij.nu`, `install/rust.nu`, `nu-plugin-image`)? My default: NO — keep them as opt-in subcommands like today.

4. **Claude Code + MCP** — install by default on a host, or only behind `--with-claude`? Today's Dockerfile defaults it ON (`INSTALL_CLAUDE=true`).

5. **Idempotency on populated hosts** — must the script be safe to re-run, skipping or upgrading what's already there? (The existing `install/topiary.nu` and `install/nushell.nu` are already written this way; the brew line and dotfiles deploy are not.)

6. **`global-claude.md` append** — this catalogs sandbox-only tools (e.g. `pbcopy` shim, mounted workspace). On a host install, do we still append it to `~/.claude/CLAUDE.md`, skip it, or replace it with a host-specific equivalent?

7. **Where the bash bootstrap installs brew on macOS** — current Dockerfile uses `NONINTERACTIVE=1` + the official curl|bash. On macOS the same script works but prompts for sudo for `/opt/homebrew`. Acceptable to let it prompt, or do you want a "brew already present? skip" guard plus an early "you'll be asked for sudo" warning?

8. **Sandbox-only bits to keep in Dockerfile** — confirm these stay Docker-only and out of `bootstrap.sh`:
   - `pbcopy` OSC52 shim → Docker-only ✓
   - `/etc/sandbox-persistent.sh` env exports → Docker-only ✓
   - apt proxy `host.docker.internal:3128` → Docker-only ✓
   - `/etc/gitconfig` build-time identity fallback → Docker-only ✓ (host already has user identity)
   - `XDG_*_HOME` env (`HELIX_RUNTIME`, `TERM`, `LANG`, ...) → Docker-only? or also exported by the host bootstrap into the user's shell rc?

9. **Future XDG prefix shape** — for the "install into an alternative dir on an existing machine" goal, which model do you prefer:
   - (a) Single env var `COZY_TARGET_HOME=/path/to/alt-home` resolved internally everywhere; user sets `XDG_CONFIG_HOME=/path/to/alt-home/.config` etc. in their shell to activate it.
   - (b) `bootstrap.sh --prefix /path/to/alt-home` flag plumbed to nu modules.
   - (c) Both — flag wins, env is fallback.

10. **PR sequence** — OK with the four-step sequence above, or do you want one big rearrangement?
